3. Storage Layer (Your Cloud Storage): Data stored in S3/ADLS/GCS, accessed via Delta Lake.

🔷 Delta Lake - The Storage Layer Foundation

What is Delta Lake?

Delta Lake is an open-source storage layer that brings ACID transactions, scalable metadata handling, and unified streaming/batch processing to data lakes. It sits on top of cloud object storage (S3, ADLS, GCS) and provides database-like reliability.

Atomicity, Consistency, Isolation, Durability - database guarantees on data lake storage

💾 Storage Formats in Databricks

Databricks supports multiple storage formats, each optimized for different use cases:

1️⃣ Delta Lake (Recommended)

🔷 Delta Lake Format

Storage: Parquet files + Transaction log (_delta_log/)

Best For: Production data pipelines, analytics, ML, any workload requiring reliability

✅ Pros:
  • ACID transactions
  • Time travel & versioning
  • Schema enforcement & evolution
  • Optimized performance (Z-ordering)
  • Concurrent reads/writes
  • Data quality checks (expectations)
❌ Cons:
  • Slightly more storage (transaction log)
  • Learning curve for optimization
  • Requires maintenance (VACUUM, OPTIMIZE)
# Create Delta table df.write.format("delta").save("/mnt/delta/customers") # With partitioning df.write.format("delta") \ .partitionBy("country") \ .save("/mnt/delta/customers") # Read Delta table delta_df = spark.read.format("delta").load("/mnt/delta/customers")

2️⃣ Apache Parquet

📦 Parquet Format

Storage: Columnar binary format, highly compressed

Best For: Read-heavy analytics, static datasets, archival storage

✅ Pros:
  • Excellent compression (70-90% reduction)
  • Fast columnar reads
  • Predicate pushdown support
  • Wide ecosystem support
  • Schema embedded in files
❌ Cons:
  • No ACID transactions
  • No schema evolution
  • Difficult to update/delete rows
  • Small file problem
# Write Parquet df.write.format("parquet") \ .mode("overwrite") \ .save("/mnt/parquet/customers") # Read Parquet parquet_df = spark.read.format("parquet") \ .load("/mnt/parquet/customers")

3️⃣ Apache Avro

📋 Avro Format

Storage: Row-based binary format with schema

Best For: Streaming data, write-heavy workloads, schema evolution

✅ Pros:
  • Great for write-heavy workloads
  • Schema evolution support
  • Compact binary format
  • Splittable for Hadoop/Spark
❌ Cons:
  • Slower for analytical queries (row-based)
  • Larger files than Parquet
  • Less efficient for columnar operations

4️⃣ Apache ORC

🗂️ ORC Format

Storage: Optimized Row Columnar format

Best For: Hive integration, Hadoop ecosystems

✅ Pros:
  • Excellent compression
  • Fast reads with Hive
  • Built-in indexes
  • Stripe-based architecture
❌ Cons:
  • Less adoption than Parquet
  • Primarily Hadoop-centric
  • Limited cross-platform support

5️⃣ CSV (Text-based)

📄 CSV Format

Storage: Plain text, comma-separated values

Best For: Data exchange, human readability, small datasets

✅ Pros:
  • Human-readable
  • Universal compatibility
  • Easy to debug
  • Simple tooling
❌ Cons:
  • No compression (huge files)
  • No schema enforcement
  • Slow to parse
  • Type inference issues
  • Not suitable for production

6️⃣ JSON

🔤 JSON Format

Storage: Text-based, hierarchical structure

Best For: Semi-structured data, APIs, nested structures

✅ Pros:
  • Nested data support
  • Schema flexibility
  • Human-readable
  • Wide ecosystem
❌ Cons:
  • Large file sizes
  • Slow parsing
  • Verbose syntax
  • Not optimized for analytics

📁 File Organization & Structure

Delta Lake File Structure

/mnt/delta/customers/ ├── _delta_log/ │ ├── 00000000000000000000.json # Transaction 0 │ ├── 00000000000000000001.json # Transaction 1 │ ├── 00000000000000000002.json # Transaction 2 │ └── 00000000000000000010.checkpoint.parquet # Checkpoint ├── part-00000-abc123.snappy.parquet # Data file 1 ├── part-00001-def456.snappy.parquet # Data file 2 ├── part-00002-ghi789.snappy.parquet # Data file 3 └── country=US/ # Partition (if partitioned) ├── part-00003-jkl012.snappy.parquet └── part-00004-mno345.snappy.parquet
Transaction Log (_delta_log/):

Every write operation creates a JSON file recording: files added, files removed, metadata changes, statistics. Checkpoint files (every 10 commits) consolidate the log for faster reads.

Partitioning Strategy

📂 Partition Example: Sales Data by Date
/sales_data/ ├── year=2025/ │ ├── month=01/ │ │ ├── day=01/ │ │ │ └── part-00000.parquet │ │ └── day=02/ │ │ └── part-00001.parquet │ └── month=02/ │ └── day=01/ │ └── part-00002.parquet └── year=2026/ └── month=01/ └── day=01/ └── part-00003.parquet
Benefits:
  • ✅ Partition pruning: Read only relevant files
  • ✅ Faster queries: Skip irrelevant partitions
  • ✅ Organized data: Logical file structure

⚡ Delta Lake Optimization Features

1. Z-Ordering (Data Skipping)

What it does:

Co-locates related data in the same set of files, enabling dramatic data skipping during queries.

-- Optimize with Z-ordering OPTIMIZE customers ZORDER BY (country, region); -- Before: Scan 1000 files -- After: Scan 10 files (99% data skipping!)

2. File Compaction (OPTIMIZE)

What it does:

Combines small files into larger, optimally-sized files (target: 1GB each) for better I/O performance.

-- Compact small files OPTIMIZE customers; -- Before: 10,000 files × 1MB = lots of overhead -- After: 10 files × 1GB = optimal performance

3. Vacuum (Cleanup Old Files)

What it does:

Removes files that are no longer referenced by the transaction log, saving storage costs.

-- Remove files older than 7 days VACUUM customers RETAIN 168 HOURS; -- Saves storage but removes Time Travel to those versions

📝 Summary & Best Practices

✅ DO Use

  • Delta Lake for all production workloads
  • Medallion architecture (Bronze/Silver/Gold)
  • Partitioning for large tables
  • Z-ordering on filter columns
  • Regular OPTIMIZE operations

❌ DON'T Use

  • CSV/JSON for production data
  • Parquet without Delta (missing ACID)
  • Over-partitioning (too many small partitions)
  • Frequent VACUUM (lose Time Travel)
  • Writing to same table from multiple clusters
